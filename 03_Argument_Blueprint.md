## 2. 逐主张的详细论据链

### 2.1 子主张 A：现有方法的跨分布泛化失效（问题必要性）

**主张陈述**：现有基于NCO的VRP求解方法在仓储异构AGV场景中面临严重的跨分布泛化失效，这一问题尚未被充分研究。

**论证结构**（三段式）：

```
A₁: NCO模型的约束紧度敏感性
  │
  │  证据: Fu et al. (NeurIPS 2025) [8] 实验表明，在CVRP容量=50上训练的AM模型，
  │        应用于容量=10时Gap扩大3-5倍，应用于容量=500时Gap扩大2-3倍。
  │        根源: 模型过拟合了特定紧度下的"容量-路线长度"权衡关系。
  │
  ├── A₂: 仓储场景天然多分布特性
  │   │
  │   │  论证: 与标准VRP基准不同，仓储场景的变化维度更多：
  │   │  - 仓库布局(grid size, 障碍物分布)
  │   │  - 任务密度(peak/off-peak)
  │   │  - AGV配比(小型:中型:大型比例)
  │   │  - AGV数量(10~100+)
  │   │  每个维度变化都构成一个新的分布，传统单模型无法覆盖。
  │   │
  │   └── A₃: 异构AGV引入额外泛化维度
  │       │
  │       │  论证: 现有泛化研究 [8][16][17] 仅考虑单车类型或同构车队。
  │       │  异构车队中，不同AGV类型存在不同容量/速度/能耗约束，
  │       │  使得约束紧度对每辆车不同，形成"内部异质分布"。
  │       │  HCVRP研究 [9] 解决了异构建模但未解决泛化问题。
  │       │
  │       └── Gap识别: 现有工作 = HCVRP建模 ∪ NCO泛化，缺失 HCVRP ∩ NCO泛化 → 本文填补
  │
  └── 结论: 仓储异构AGV场景的跨分布泛化是一个真实存在且未被解决的研究空白
```

**可能攻击点与防御**：

| 攻击 | 防御 |
|------|------|
| "为什么不直接用传统方法（LKH3）？每次重新求解就不存在泛化问题" | LKH3在大规模问题上的求解时间是小时级，无法满足仓储实时调度需求（秒级）。实时性是仓储场景的硬约束。 |
| "为什么不针对每个仓库单独训练一个模型？" | 训练一个NCO模型需要百万级实例和数天GPU时间。每个仓库单独训练在成本和效率上不可行，这恰恰是泛化研究要解决的问题。 |
| "异构AGV不就是HCVRP吗？已有研究 [9] 解决了" | [9] 解决了同分布下的异构建模，但未探讨跨分布泛化。本文的贡献是将异构建模与泛化机制统一在一个框架内。 |

### 2.2 子主张 B：MEHA三组件各自解决泛化的不同维度（方法有效性）

**主张陈述**：MEHA框架通过Multi-Expert+TGate、HeteAttn、可行掩码三个组件，分别从分布级、AGV异构级和仓储约束级三个层面解决泛化问题。

#### B₁: Multi-Expert + TGate → 分布级泛化

```
直觉: 单一模型需要同时拟合多个分布，导致"平均化"策略——在各分布上都次优。
     多专家机制允许不同专家特化于不同分布，门控网络根据场景特征动态路由。

证据链:
  ┌── 理论支撑: Mixture of Experts (MoE) 在大语言模型中已被证明能有效处理
  │   多域数据，避免不同域间的负迁移 (Shazeer et al., 2017)。
  │
  ├── 直接前驱: Fu et al. [8] 提出 multi-expert module 解决CVRP约束紧度泛化。
  │   本文将其从单一约束维度扩展到多维仓储场景约束。
  │
  └── 消融验证: 实验5.3.1将证明移除Expert后跨分布退化率回升至基线水平。
```

**门控机制设计的论证**：

```
输入特征选择:
  - AGV数量: 影响动作空间规模和协调复杂度
  - 任务密度 = |tasks| / (grid_w * grid_h): 影响路径拥堵程度
  - 平均容量利用率 = E[demand_per_task * tasks_per_agv / agv_capacity]: 影响容量约束紧度
  - AGV类型熵 = -Σ p(type) * log p(type): 影响异构程度
  
  设计理由: 这4个特征在推理时可直接从问题实例计算，无需额外标注，
           且理论上有足够信息量区分分布类型。
```

#### B₂: HeteAttn → AGV异构级泛化

```
直觉: 标准自注意力将所有节点无差别对待。但在异构AGV场景中：
      - 同类型AGV之间需要协调（争抢同类任务）
      - 跨类型AGV之间需要互补（大型AGV处理重任务，小型AGV处理轻任务）
      无差别注意力混淆了这两种不同性质的交互。

设计原理:
  ┌── 注意力分组: 将H个注意力头分为两组
  │   - 组1 (H₁个头): 仅计算同类型节点间的注意力 (mask掉跨类型边)
  │   - 组2 (H₂个头): 计算所有节点间的注意力
  │
  ├── 掩码机制: 对组1的每个头，构建类型掩码矩阵 M ∈ {0, -∞}^{N×N}
  │   M[i,j] = 0 if type(i) == type(j) else -∞
  │   Attention = softmax(QK^T/√d + M) · V
  │
  └── 泛化收益: 当AGV配比变化(D_test4)时，组1的同类型头不受影响，
               组2的跨类型头已完成充分训练，组合泛化能力更强。
```

#### B₃: 集中式解码 + 可行掩码 → 仓储约束合规

```
直觉: 标准AM/POMO的解码器一次只输出一个动作（一辆车的下一个节点）。
     仓储场景需要同时协调多辆AGV，且必须保证约束合规。

设计理由:
  ├── 集中式输出: 每个解码步输出M个节点选择（M=AGV数量），
  │   而非逐个输出。这避免了序列化导致的协调延迟。
  │
  └── 三重掩码保证可行性:
      - 容量掩码: task.demand > agv.remaining_capacity → mask
      - 电量掩码: dist(agv.pos, task.pos) > agv.remaining_range → mask
      - 冲突掩码: 同一步内，已被其他AGV选择的任务 → mask
      
      掩码后的动作空间始终是可行子集，无需后处理修复。
```

**三组件协同逻辑**：

```
输入实例
  │
  ├──[HeteAttn 编码]──→ 异构感知嵌入 h_enc
  │
  ├──[Tightness Encoder]──→ 场景紧度编码 z_tight
  │
  ├──[TGate 路由]──→ 专家权重 w
  │
  ├──[Multi-Expert 处理]──→ h_fused = Σ w_k * Expertₖ(h_enc)
  │
  └──[可行掩码解码]──→ 合规的联合动作序列
```

### 2.3 子主张 C：泛化不牺牲基础性能（实用价值）

```
主张陈述: MEHA在实现跨分布泛化的同时，在同分布条件下保持与SOTA相当的基础性能。

论证:
  ┌── 逻辑: 多专家模块的特化设计使得在训练分布上至少有一个专家是"主场作战"，
  │   其性能不应低于单模型。门控会自然赋予该专家高权重。
  │
  ├── 对照: 引入额外组件（Multi-Expert, TGate, HeteAttn）可能增加过拟合风险。
  │   需要通过以下实验消除疑虑：
  │   - 5.2.1 同分布性能对比
  │   - 5.3.1 w/o Expert 变体对比
  │
  └── 预期结果: 同分布Gap ≤ POMO ± 0.5%，证明额外组件未损害基础能力
```

## 5. 论证压力测试

### 5.1 最强反驳：同分布测试结果不如POMO

```
风险: 如果同分布(5.2.1)结果显著差于POMO（Gap > 2%），
      则主张C被证伪，论文核心论点受损。

缓解措施:
  1. 架构设计时保留与POMO的兼容性：HeteAttn可退化为标准注意力(所有头为跨类型)
  2. 如果同分布确实较差，可设计"MEHA-Lite"变体（w/o Expert, 同分布用），
     仅在跨分布时激活全功能
  3. 在§6讨论中诚实分析性能trade-off
```

### 5.2 最强反驳：多专家的分布特化不明显

```
风险: 如果门控权重可视化显示所有专家权重均匀（均≈1/K），
      则证明多专家未产生实质特化，仅仅是参数冗余。

缓解措施:
  1. 在训练阶段加入负载均衡损失(Load Balancing Loss)：
     L_lb = K * Σ f_k * w_k，其中f_k是分配到专家k的token比例
     该损失激励门控产生非均匀分布
  2. 对比实验：固定均匀权重 vs 学习门控，验证门控的必要性
  3. 如果权重确实均匀，则转向分析均匀权重是否仍有收益（ensemble effect）
```

### 5.3 最强反驳：跨分布改善仅来自数据增强，非方法贡献

```
风险: 审稿人可能认为跨分布泛化提升仅因为"多分布采样训练"，
      而非Multi-Expert架构本身的贡献。

关键消融:
  - 对比 1: 单模型 + 单分布训练 (AM baseline)
  - 对比 2: 单模型 + 多分布采样训练 (数据增强baseline)
  - 对比 3: MEHA (Multi-Expert + 多分布采样)
  
  如果对比2已经接近对比3，说明方法贡献有限。
  必须证明: Gap(对比3, 对比2) 在统计上显著 > 0

论文中需要在§5.3.1加入此baseline（"数据增强baseline"），这是最强的消融。
```
